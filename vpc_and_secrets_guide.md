# VPC 변경에 따른 영향 및 시크릿 관리 가이드

새로운 VPC를 만들고 인프라를 이전할 때, 어떤 것이 변하고 어떤 것이 그대로 유지되는지 정리해 드립니다.

---

### 1. 무엇이 바뀌나요? (What Changes?)

| 항목 | 상태 | 변화 내용 |
| :--- | :--- | :--- |
| **네트워크 주소** | **변경됨** | 서비스가 돌아가는 '동네'가 바뀝니다. 엔드포인트 주소가 새로 생성됩니다. |
| **보안 그룹 (SG)** | **변경됨** | 구 VPC의 보안 그룹은 새 VPC에서 쓸 수 없습니다. 새로 만들어야 합니다. |
| **데이터베이스 정보** | **변경됨** | SQLite 파일 방식에서 RDS(Network 방식)로 바뀝니다. 호스트 주소가 생깁니다. |
| **GitHub Secrets** | **추가됨** | `DB_HOST`, `DB_PASSWORD` 등 RDS 연결을 위한 정보들이 추가로 필요합니다. |

### 2. 무엇이 유지되나요? (What Stays?)

*   **Django SECRET_KEY**: 소스 코드 레벨의 암호화 키이므로 네트워크 환경과 무관하게 유지됩니다.
*   **AWS Access/Secret Key**: 내 AWS 계정 전체에 대한 권한이므로 VPC와 상관없이 동일합니다.
*   **OpenAI API Key**: 외부 서비스 키이므로 그대로 사용합니다.

---

### 3. 전체 흐름 (End-to-End Flow)

1.  **[Foundation]** **VPC 마법사**로 '멀티 AZ(2개 동네)' 환경을 한 번에 구축합니다.
2.  **[Database]** 방금 만든 신규 VPC 안에 **RDS**를 생성합니다. (에러 없이 성공 가능)
3.  **[Code]** Django가 SQLite 대신 이 RDS를 바라보도록 `.env` 파일을 수정합니다.
4.  **[Container]** 소스 코드를 구워서 **AWS ECR**이라는 창고에 저장합니다.
5.  **[Launch]** **ECS(Fargate)**가 이 창고에서 코드를 꺼내 신규 VPC 위에서 실행합니다.

---

### 4. 구체적인 체크리스트 (사용자님 확인용)

새 VPC를 구축하더라도 사용자님이 **지금 당장** 바꾸실 "비밀 키"는 없으니 안심하세요! 나중에 설정할 항목들만 미리 눈여겨보시면 됩니다.

*   [ ] **VPC 생성**: `VPC and more` 옵션으로 2개의 가용 영역 확보
*   [ ] **RDS 생성**: 위에서 만든 VPC 선택하여 생성
*   [ ] **보안 그룹**: RDS용 보안 그룹에서 **포트 5432**(PostgreSQL)가 열려 있는지 확인
*   [ ] **GitHub Secrets 업데이트**:
    *   `DB_HOST`: RDS 엔드포인트
    *   `DB_NAME`: `practice_db`
    *   `DB_USER`: `postgres`
    *   `DB_PASSWORD`: (사용자님이 설정하신 암호)

---

**결론**: VPC를 새로 만드는 것은 "새 집으로 이사"하는 것과 같습니다. 이사를 간다고 해서 개인 비밀번호(`SECRET_KEY`)가 바뀌진 않지만, **새로운 집 주소(`DB_HOST`)**는 확인해서 친구들(GitHub Actions)에게 알려줘야 하는 것과 같습니다.
